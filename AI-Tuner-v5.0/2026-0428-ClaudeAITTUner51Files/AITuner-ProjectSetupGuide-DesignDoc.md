# AITuner — Project Setup Guide
## Feature Design Document
**Date:** April 28, 2026
**Status:** Design — not yet spec'd for Cursor
**Feature name:** Project Setup Guide
**Entry point:** Home screen (fifth card) +
  Model Room tab (alongside About / Tune / Decode)

---

## What This Feature Is

AITuner currently helps users tune AI behavior —
levers, personas, generated prompts. But it doesn't
help users set up the *container* the conversation
lives in. The project workspace. All three zones
working together.

The Project Setup Guide fills that gap. It walks a
user through configuring their entire working
environment — not just a system prompt, but the
knowledge layer, the behavioral layer, and an
understanding of the memory layer — before they
start working.

Output: three deliverables, not one.
1. A Knowledge Template — ready to upload as a file
2. Instructions Content — formatted for the Instructions field
3. A Memory Guide — how to work with what builds over time

---

## The Core Distinction This Feature Must Teach

This is the most important thing the feature communicates.
Most users don't know it exists.

### Instructions field vs Prompt box

| | Instructions field | Prompt box |
|---|---|---|
| When set | Once per project | Every message |
| What it does | Persistent behavioral rules | Task direction |
| Persists | Always active | That exchange only |
| Zone | Zone 1 — static layer | Zone 1 — dynamic layer |
| Analogy | The standing orders | The daily briefing |
| AITuner generates | Yes — lever output goes here | Yes — task prompts separately |

The Instructions field is the behavioral container.
The prompt box is how you communicate within it.
They are not interchangeable.
Most users treat them as the same thing.
That's why their configurations don't hold.

## Three Sections — One Per Zone

---

### Section 1: Knowledge
*Zone 1 — Static layer*
*"What to put in your files"*

#### What it does

Helps users build a structured knowledge base —
the files that give the AI context about who they
are and what they're working on. Separate from
behavioral instructions. Never mixed.

#### Why separation matters

When knowledge and behavior are mixed in the same
document, the model uses personal context to
*generate* responses instead of following
behavioral rules. Most AI instruction failures
start with this entanglement.

Two separate files. Always.

#### Guided checklist — what belongs in knowledge

**Project documents**
- Project briefs, scopes, requirements
- Research, reference material, source documents
- Decisions already made and why
- Key constraints and non-negotiables

**Personal context**
- Name, role, expertise level
- How you think and work
- Communication preferences
- What you already know (so the AI doesn't explain it)

**Named entities**
- Key people and their roles
- Companies, clients, organizations
- Products, systems, terminology specific to your world
- Relationships and dynamics that matter

**Key frameworks**
- Mental models you use
- Methodologies, processes, approaches
- Reference systems (Gottman, Clear, etc.)
- Ways of thinking you want the AI to mirror

**Historical context**
- Relevant past work
- Previous decisions and their outcomes
- What's been tried and didn't work
- The context that makes current work make sense

#### Output: Knowledge Template

A structured markdown file the user fills in.
Not a prompt — a knowledge document.
Formatted the way Claude and Grok expect files.

```markdown
# [Project Name] — Knowledge Base
## Generated by AITuner · [Date]

---

## About Me
Name:
Role:
Expertise level:
How I work best:
What I already know (don't explain):

---

## This Project
What it is:
Current status:
Key decisions already made:
Key constraints:

---

## Key People & Entities
[Name / Role / Relevant context]

---

## Frameworks I Use
[Framework / How I apply it]

---

## Historical Context
[What's happened / What's been tried / What matters]

---

## Terminology
[Term / What it means in this context]
```

User fills this in. Downloads it. Uploads to their
project's files/sources section.

Platform-specific note: Show where to upload in
Claude vs Grok vs ChatGPT with annotated screenshots.

### Section 2: Instructions / Behavior
*Zone 1 — Static behavioral layer*
*"What goes in the Instructions field"*

#### What it does

Takes the AITuner lever configuration and formats it
specifically for the Instructions field — not for
copy-pasting into a chat, but for persistent
behavioral configuration of a project.

#### The key distinction — explained plainly

**Instructions field:**
> "This is standing orders. Set once. Every
> conversation in this project starts with these
> rules already active. You don't have to repeat
> them. They're always there."

**Prompt box:**
> "This is what you say in the moment. Task-specific.
> The instructions are already running — you just
> tell it what you need right now."

**Analogy that works:**
The Instructions field is like configuring a new
employee before their first day — here's how we
work, here's what I expect, here's my style.
The prompt box is what you say to them each morning.

#### What goes in Instructions

Not everything — only what should be *always true*
regardless of the task:

✓ Include:
- Tone and communication style rules
- Format preferences (bullets vs prose, length, headers)
- What to never do (affirmations, disclaimers, hedging)
- Role definition (peer, advisor, collaborator)
- Knowledge assumptions (what to never explain)
- Hard behavioral constraints

✗ Don't include:
- Task-specific instructions (those go in the prompt)
- Information about current projects (that's knowledge files)
- Anything that's only true sometimes

#### The conflict resolution rule

When instructions conflict with each other —
"be concise" vs "be thorough" — the model
resolves inconsistently. Declare a hierarchy:

> "When rules conflict: [primary rule] takes
> precedence over [secondary rule]."

AITuner generates this automatically based on
lever priority settings.

#### Output: Instructions Content

Generated from the lever configuration.
Formatted for the Instructions field.
Structured in tiers:

```
TIER 1 — ROLE
[What I am / What I'm not — 3 sentences max]

TIER 2 — HARD STOPS
[Never do these — short list, no overlap]

TIER 3 — BEHAVIORAL RULES
[How to communicate, format, prioritize]

TIER 4 — KNOWLEDGE ASSUMPTIONS
[What to assume I already know]

CONFLICT RESOLUTION
[When rules conflict, X takes precedence]
```

Platform-specific formatting note:
Claude Instructions field supports markdown.
Grok Instructions field — plain text preferred.
ChatGPT Custom Instructions — two separate fields
(about you / how to respond). Map accordingly.

### Section 3: Memory
*Zone 2 — Emergent layer*
*"What builds over time"*

#### What it does

Explains the memory layer — what it is, how it
accumulates, that only the user sees it, and
how to work with it deliberately rather than
letting it drift.

#### Plain-English explanation

Memory is what the AI learns about you through
your conversations — not from files you upload,
but from what it observes over time. It builds
automatically. It's private. You can read it,
edit it, and delete it.

The important thing to understand: memory is
Zone 2. It develops through use. It's not
something you set once — it evolves. And it
can drift in ways that work against you if
you don't manage it.

#### How memory works per platform

**Claude (Projects):**
- Project memory builds automatically after a few chats
- Shows in the project sidebar — "Memory" panel
- Labeled "Only you" — private
- Can be edited directly in the sidebar
- Prompt command: "What do you remember about me?"

**Grok:**
- Memory builds across conversations
- Less visible than Claude — no dedicated panel
- Prompt command: "What have you learned about me?"
- Can be cleared in settings

**ChatGPT:**
- Explicit memory system — shows what it's saved
- Can be viewed and edited in settings
- Prompt command: "What do you remember?"
- Toggle on/off per conversation

**Gemini:**
- Memory through Google account integration
- Less explicit than Claude/ChatGPT
- Varies by product surface

#### Memory hygiene — recommended practice

Every 2-4 weeks, or when something significant changes:

1. Ask: "What do you remember about me and this project?"
2. Review what it says
3. Correct anything outdated or wrong
4. Delete entries that no longer apply
5. Add context that's missing: "Remember that [X]"

The prompt commands that work:
- "What do you remember about me?"
- "Update your memory — [X] is no longer true"
- "Remember that [X]"
- "Forget [X]"
- "Clear everything you remember about [topic]"

#### Memory starter suggestions

When beginning a new project, seed the memory
deliberately rather than waiting for it to
accumulate randomly:

> "Remember that I'm working on [project].
> Remember that my preferred communication
> style is [X]. Remember that I already know
> [Y] — you don't need to explain it."

#### The drift warning

Memory can work against you if it accumulates
outdated context. A project that evolves
significantly may have old assumptions in memory
that pull the AI toward stale behavior.

Rule: when the project changes significantly,
audit the memory. Don't let Zone 2 lag behind
Zone 1.

## UX Flow — How the Feature Works

### Entry points

**Home screen — fifth card:**
```
Set up a project
Configure your full working environment —
knowledge, behavior, and memory — before you start.
```

**Model Room — fourth tab:**
```
About | Tune | Decode | Setup
```
Setup tab inside the room = model-specific setup.
The Claude Setup knows Claude's Instructions field format.
The Grok Setup knows Grok's format.

### Flow within the feature

**Step 0 — What you're building**
Brief explanation of the three zones.
"You're about to set up three things. Each one
does something different. None of them should
be mixed."

Visual: the three-zone diagram, simplified.

**Step 1 — Knowledge**
Guided checklist. User answers questions.
Template generates as they answer.
Download button at the bottom.
Platform-specific upload instructions.

**Step 2 — Instructions / Behavior**
Two sub-steps:
  2a — Load your lever configuration
       (or quick-configure if not done yet)
  2b — Review generated Instructions content
       Formatted for their platform.
       Copy button.
       Platform-specific paste instructions.

**Step 3 — Memory**
Education only — no generation.
Explanation of how memory works.
Prompt commands they can use.
Starter suggestions.
"Memory builds through use — check back in a
few weeks and we'll help you audit it."

**Step 4 — You're set up**
Summary of what was built:
- Knowledge Template: downloaded ✓
- Instructions: copied ✓
- Memory: understood ✓

"Now start your project. Come back to AITuner
when you want to tune the behavior or decode
a prompt."

## Platform-Specific Notes for Cursor

Different platforms expose the three zones
differently. The feature needs to know which
platform the user is setting up for and adjust:

| Zone | Claude | Grok | ChatGPT | Gemini |
|------|--------|------|---------|--------|
| Instructions | Project Instructions field | Project Instructions field | Custom Instructions (2 fields) | System instructions |
| Knowledge files | Project Files (+ upload) | Project Sources (+ attach) | Custom GPT knowledge | Gemini extensions |
| Memory | Project Memory panel | Conversation memory | Memory settings | Google account |
| Memory visible | Yes — sidebar panel | Partial | Yes — settings | Partial |
| Memory editable | Yes — directly | Via prompts | Yes — settings | Limited |

The setup flow should show annotated screenshots
of where each element lives in each platform's UI.
Use the screenshots John provided as source material.

## What This Feature Changes About AITuner

Before this feature:
AITuner generates a system prompt.
User copies it. Pastes it somewhere. Hopes it works.

After this feature:
AITuner builds a complete working environment.
Knowledge file. Instructions content. Memory guide.
The user understands what each one does and why
they're separate.

This is the difference between handing someone
a key and showing them the whole house.

---

## Connection to the Three Zones Framework

This feature IS the three zones framework made
into a product. Each section maps directly:

- Knowledge → Zone 1 (static direct input)
- Instructions → Zone 1 (static behavioral layer)
- Memory → Zone 2 (emergent layer)
- Zone 3 (fixed) → explained in model rooms,
  not configured here — because you can't configure it

The feature doesn't pretend you can control Zone 3.
It teaches you what Zone 3 is and why you're hitting
a wall when you are. That honesty is the product.

## Design Decisions — Resolved April 28, 2026

All five open questions answered. Cursor can
implement Stop 10 without further clarification.

1. **Single guided flow** — one flow, three steps,
   sequential. Not three separate tools.

2. **Model-specific** — the setup flow knows which
   platform it's configuring for from the start.
   Platform context is set when entering from a
   model room. Claude setup knows Claude's format.
   Grok setup knows Grok's format.

3. **Markdown** — Knowledge Template outputs as
   a markdown file. Most portable. Ready to upload
   directly to project files/sources in any platform.

4. **Prompt commands only** — Memory section shows
   platform-specific prompt commands and a one-paragraph
   explanation. No full hygiene education. Keep it fast.

5. **Model Room Tab** — Setup lives as a fourth tab
   inside each model room: About | Tune | Decode | Setup.
   Entry point is model-specific from the start.
